User reported splat outputs look bad: **floaters/fog + ghosting/doubling**. Root cause is NOT PPISP — it's that recent commits optimized for benchmark SPEED and starved quality. Evidence from job `270479fdb8cd`: 32 frames, **~105s training** on a 4090, densified to 850k Gaussians (22% near-transparent floaters), floater-cleanup pass self-skipped ("would remove 100% of survivors" = no coherent connected surface = diffuse geometry).

**Why:** the dominant 3DGS quality levers are view coverage (→ fewer floaters, better pose graph) and pose accuracy (→ less ghosting). PPISP is only a per-frame exposure/WB polish and, because its corrections aren't applied at export, can even slightly *hurt* (global color shift). Don't reach for PPISP to fix geometry.

**How to apply:** balanced quality config applied 2026-06-23 in `worker/config.py`: `default_max_frames` 50→100, `dust3r_alignment_iters` 300→500, `dust3r_swin_window` 7→**8** (started at 9, dialed back after OOM), `densify_grad_thresh` 0.0002→0.0003, `cleanup_link_ratio` 5→7, and PPISP disabled via new `ppisp_enabled=False` flag (gated in `train.py`). Next escalation lever if ghosting persists: `dust3r_amp=False` (fp32 poses) — held back to stay in budget.

**Dead-settings bug (found 2026-06-23):** `settings.default_max_frames/default_training_iterations/default_resolution` were defined but read NOWHERE — the real defaults were hardcoded in `app.py` `create_job` (`Form(default=40, le=80)`, iters 7000) and `queue_client.py` (`config.get("max_frames", 40)`). So the benchmark "use defaults" toggle silently ran 40 frames/7000 iters and the whole quality config never executed (poses stayed ~120s, results unchanged). Fixed: both entry points now fall back to `settings.default_*`, and the `le` cap raised 80→200 (+ benchmark frames input max 80→200). Restart applied the fix; endpoint now uses 100 frames/10000 iters on omitted fields.

**Pose-stage VRAM wall (confirmed 2026-06-24):** DUSt3R global alignment optimizes ALL views + pairs jointly on the GPU and CANNOT be streamed. 100 frames (→80 after blur) at swin-8 = 1280 pairs filled all 24GB and spilled 3.4GB into Windows shared memory (no crash, just thrashes at 100% — pathologically slow). The GPU was otherwise FREE (other containers idle), so this is intrinsic, not contention. Peak VRAM ~ pairs = 2*frames*swin_window. Safe ceiling on 24GB ≈ 768 pairs → set `default_max_frames=80` (~64 after blur) + `dust3r_swin_window=6`. UPDATE: even 64 frames/768 pairs at 512 res maxed 24GB (per-image depthmaps don't scale down with pair count). Tried `dust3r_resolution=384` to keep 64 frames — REVERTED: poses degraded badly ("worst result yet"). 512_dpt model needs 512; resolution is NOT a usable lever. So 512 res hard-caps frames at ~44-50 on 24GB. Final safe config: `dust3r_resolution=512`, `default_max_frames=56` (~44 after blur, 528 pairs, ~18GB). This is only modestly more coverage than the original 40 — frame count can't be the big win on this hardware. The untested question: whether the NON-frame fixes (PPISP off, alignment 500, densify 0.0003, pose-opt, swin-6 vs the original PPISP-on/align-300/densify-0.0002-overgrowing-850k) improve quality in a clean COMPLETED run. If 512/44-frame still looks bad, bottleneck is DUSt3R pose quality or the capture itself, not config — next hammer would be a different pose backend (COLMAP) or more VRAM. Streaming-frames fix only helped TRAINING, not the pose stage.

**Floater removal was a NO-OP the whole time (fixed 2026-06-24):** the cleanup floater pass (connected-components over a kNN graph) self-skipped every run ("would remove 100% of survivors") because near-coincident split Gaussians make the link radius tiny and the graph shatters — so it removed 0 floaters while the user saw "tons of floaters/random splats." Replaced with statistical outlier removal (SOR) in `cleanup.py`: flag Gaussians whose mean kNN distance > `cleanup_sor_dist_ratio`(=20) × the median, bounded by `cleanup_floater_max_remove_frac`(=0.15). MAD-based threshold failed (dense core median 0.003 is pathologically tight → flagged 30%); median-MULTIPLE works. Validated on real PLY 639bba3ec73d: removes 6.5% that sit far from center (median dist 3.96 vs 2.95 kept, tail to 11.3 vs object radius ~5) = genuine floaters, object intact. Removed dead config `cleanup_link_ratio`/`cleanup_min_component_frac`. Lower ratio = more aggressive.

**Earlier memory caveat:** the frames 50→100 jump OOM'd the 4090 (which is SHARED with `comfyui`/`puffertank`/`triposplat-worker` containers). Fix applied: `train.py` `_load_images_as_tensors` now keeps frames on CPU (pinned) and moves only the per-step batch to GPU via `.to(device, non_blocking=True)`, so training VRAM is independent of frame count. Pose-stage (DUSt3R global aligner) VRAM still scales with frames×window — 100 frames is aggressive on a shared card; free comfyui's VRAM or lower frames if it OOMs again. The `splatapp-worker` container was REMOVED (gone from `docker ps -a`) and must be recreated to test. Changes uncommitted. See [[worker-runs-from-host-bind-mount]].
