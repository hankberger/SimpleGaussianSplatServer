The worker's quality AND memory problems both trace to ONE component: the pose backend, **DUSt3R**. Confirmed 2026-06-24 when the user ran COLMAP→LichtFeld Studio on the SAME input and got dramatically better results that fit the 4090 easily.

**Why DUSt3R is the bottleneck:**
- Quality: DUSt3R poses are approximate (neural pointmap regression + weak global alignment), never optimized to reprojection error. 3DGS quality is dominated by pose accuracy → DUSt3R's error shows as ghosting/blur/floaters. COLMAP = geometric SfM + bundle adjustment, sub-pixel poses (the gold standard 3DGS was designed around). Also: COLMAP sparse init is multi-view-verified (DUSt3R dense init is noisy → seeds floaters from step 0); COLMAP solves intrinsics + lens distortion precisely (DUSt3R approximates, ignores distortion).
- Memory: DUSt3R's global alignment optimizes ALL views+pairs jointly on GPU (+3.5GB ViT) = the 24GB wall behind every OOM this session. COLMAP is CPU-bound (feature matching + BA), barely touches GPU; training itself is cheap. So COLMAP also removes the frame ceiling (can feed 150-200 frames).

**Meta-lesson:** all the session's config tuning (frames, swin_window, dust3r_resolution, pose_opt, SOR floater cleanup in [[splat-quality-root-cause]]) was treating SYMPTOMS of approximate poses. The fix is replacing the pose stage, not tuning it.

**IMPLEMENTED 2026-06-24 (option A: COLMAP primary + DUSt3R fallback, kept gsplat trainer):**
- `worker/pipeline/poses_colmap.py` (new): `estimate_poses_colmap(frame_paths)` via `pycolmap` (v4.0.4). extract_features (camera_mode=SINGLE — one phone camera) → match (exhaustive) → incremental_mapping → extract c2w poses (invert cam_from_world; it's a METHOD in pycolmap 4.x: `img.cam_from_world().matrix()`), K (cam.calibration_matrix(), already at frame res), sparse points+colors. Returns registered SUBSET of frames.
- `worker/pipeline/poses.py`: `estimate_poses(frame_paths, config)` is now a DISPATCHER — tries COLMAP, falls back to DUSt3R (`_estimate_poses_dust3r`) on any exception or if registered < `colmap_min_registered_frac`(0.5)×frames. Shared `_normalize_scene` (centroid-center, 95th-pct radius→5) applied to both. Intrinsic rescale moved INTO dust3r path (returns at training res); removed from app.py.
- `app.py`: pose stage returns 6-tuple `(poses, intrinsics, points, colors, used_frame_paths, backend)`; rebinds frame_paths to the registered subset so training stays aligned; stage detail shows backend.
- Config: `colmap_enabled=True`, `colmap_min_registered_frac=0.5`, `colmap_matcher="exhaustive"` (sequential FAILED to register — 64 frames sampled ~1/sec have too-large baselines; exhaustive registered 64/64). `requirements.txt`: `pycolmap>=0.6.1`.
- VALIDATED on real frames (job 0bc42477bb94, 64 frames): exhaustive → 64/64 registered, valid poses (rot det=1), 12884 points, K correct (cx=216,cy=384 for 432x768). Sequential = "no good initial pair".
- KEY GOTCHA #1: COLMAP wants EXHAUSTIVE matching at our frame counts + needs decent overlap; the DUSt3R-tuned sparse uniform sampling is borderline. If COLMAP starts failing (→silent DUSt3R fallback, bad output again), check frame overlap/resolution.
- KEY GOTCHA #2 — DISTORTION (fixed 2026-06-24): first COLMAP result was "mush / doesn't look like anything." Cause: phone lenses have real radial distortion (COLMAP's default SIMPLE_RADIAL model estimates it), but we extracted only the pinhole `calibration_matrix()` and trained with the pinhole gsplat rasterizer on the still-DISTORTED images → Gaussians can't reconcile distorted pixels with pinhole projection → collapse. The classic COLMAP→3DGS pipeline ALWAYS undistorts; we skipped it. User's tell: they had to enable 3DGUT in LichtFeld (3DGUT models distortion in the projection instead of undistorting). FIX: `poses_colmap.py` runs `pycolmap.undistort_images(undist_dir, rec_dir, image_dir)` after mapping, loads the PINHOLE undistorted reconstruction, returns UNDISTORTED image paths as used_frame_paths. Lesson: any pinhole 3DGS trainer needs undistorted input; 3DGUT avoids undistortion by modeling distortion in projection.

**KEY GOTCHA #3 — CAMERA MODEL (fixed 2026-06-25): "still looks distorted" after undistortion.** Cause: pycolmap default camera model is SIMPLE_RADIAL (single k1) — UNDER-corrects phone lenses, leaving residual radial(k2)+tangential(p1,p2) distortion. User's working COLMAP recipe used OPENCV model (k1,k2,p1,p2). FIX: `colmap_camera_model="OPENCV"` set via `reader_options.camera_model` in extract_features. Matched the rest of user's proven recipe too: `colmap_matcher="sequential"` (overlap 10 default — fast, right for dense video; exhaustive is O(n^2) slow on CPU SIFT), `default_max_frames=120` (~2 fps, was 56), `dust3r_max_frames=50` (dispatcher subsamples the 120-frame set before the DUSt3R fallback to avoid OOM). User's recipe metrics: 128 frames @2fps, 128/128 registered, OPENCV 1 shared camera, 59402 pts, 0.50px reproj err. Validated OPENCV registers+undistorts (exhaustive on sparse test frames; sequential needs the dense 120-frame extraction).

**RESOLVED — it was the MATCHER, not training (2026-06-25).** The "distorted" output was a BROKEN COLMAP reconstruction from `sequential` matching: sequential registered only 59/96 frames with 4018 points (drifted/partial poses) → distorted; `exhaustive` registered 52/52 with 11008 points → clean, and training loss dropped 0.17→0.065. WHY sequential failed for us but worked in the user's standalone recipe: our blur filter (`blur_drop_ratio`) drops frames UNEVENLY, breaking sequential matching's even-temporal-spacing assumption (large baselines between "neighbors"); exhaustive is order-independent. FIX (initial): `colmap_matcher="exhaustive"`. FINAL VERDICT (2026-06-26): `colmap_matcher="exhaustive"` PERMANENTLY. Sequential was tried twice and abandoned both times — even after the decouple (below) it gave WEAK reconstructions on the user's real capture: 93/120 registered but only 7685 points + many "no good initial pair"/Cholesky/Gauge failures → distorted, vs exhaustive's clean 52/52 + 11008 points = "a lot better". Sequential is fundamentally fragile for THIS pipeline: 768px frames have few SIFT features AND a blurry frame between two sharp ones breaks its consecutive-match chain (delete blurry → uneven gaps break it; keep blurry → blur links break it). Exhaustive tries all pairs → robust to both. Cost: O(n^2) ~5-7min CPU SIFT at 120 frames. DO NOT try sequential again without first raising COLMAP frame resolution (more features).

The DECOUPLE infrastructure was kept (harmless with exhaustive, gives COLMAP the full frame set): 2026-06-25 change so sequential could work — `filter_blurry_frames` no longer deletes soft frames (they stay on disk); `_run_frame_extraction` returns `(all_frames, sharp_frames)`; `estimate_poses(all_frames, sharp_frames, config)` runs COLMAP on the FULL evenly-spaced set (sequential matches fine) then filters the registered cameras to the sharp subset by filename for training (keeps full sparse cloud as init). DUSt3R fallback runs on sharp_frames (subsampled to dust3r_max_frames). `colmap_matcher="sequential"` again. If a capture's sequential reconstruction is poor it auto-falls-back to DUSt3R (no auto-switch to exhaustive — set colmap_matcher=exhaustive manually if needed). The pose_opt-off-for-COLMAP change was NOT the visible fix but is kept (correct not to refine accurate poses). Cost: exhaustive is O(n^2) on CPU SIFT, ~slow at 120 frames but acceptable. Debug-fast preset = 64 frames / 3000 iters; quality = 120 / 10000 (current).

**BISECTED TO TRAINING (2026-06-25) [superseded by the matcher finding above]:** added a downloadable COLMAP dataset (`/api/v1/jobs/{id}/dataset`, benchmark button, `colmap_save_dataset=True`; zip = images/+sparse/0/ undistorted + distorted/ mirror; cleanup keeps .zip). User loaded OUR dataset into LichtFeld → looks FINE (just tilted = COLMAP's arbitrary up-axis, NOT a bug). So COLMAP output is good; the defect is in OUR gsplat training. PRIME SUSPECT + FIX: `train.py` camera pose optimization (pose_opt) was ON — added for DUSt3R's approximate poses, but refining COLMAP's already-accurate poses lets cameras drift and WARPS the scene. Fix: `train_gaussians(optimize_poses=...)` param; app.py sets `refine_poses = settings.pose_opt_enabled and backend=="dust3r"` — pose-opt now OFF for COLMAP, still ON for DUSt3R fallback. Threaded `backend` from the pose dispatcher. If still distorted after this, next suspects: rasterizer convention / our hand-rolled densification vs LichtFeld. Verified NOT the bug: _normalize_scene (valid similarity transform), intrinsics (COLMAP K used directly, override loop doesn't fire), pose round-trip (c2w=inv(w2c) then back = identity → viewmats are COLMAP cam_from_world, correct for gsplat).

**OUTPUT-PATH BUG (fixed 2026-06-25):** undistorted frames live at job_dir/colmap/undistorted/images/, so train.py's `output_dir = frame_paths[0].parent.parent` wrote PLY/splat/preview into that subdir → cleanup_job_dir (only protects job_dir) deleted them → result/preview 404. FIX: train_gaussians takes explicit `output_dir`, threaded job_dir from both pipeline paths in app.py.

**Persistence:** pycolmap was added to requirements + image rebuilt (`docker build -t splatapp-worker:latest .`, log `build_colmap.log`). DUSt3R's only edge is hard captures (few images, wide baseline, no texture) — the fallback covers those.
